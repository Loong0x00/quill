### [P0-4] 偏离 #2 env LANG=C 包装决策 ✅ (技术陈述需修正, 但决策正确)

**这是 T-0307 第二个关键判断点**, 我独立 verify 写码的 "Rust 1.79+ unsafe set_var" 主张, 不直接接受。

**writer 主张** (commit body / module doc line 36-40): "Rust 1.79+ 显式把 `set_var` 标 `unsafe`, 因为它非线程安全"

**独立 verify** (rustc 1.92.0 + 项目 edition 2021):
```rust
// /tmp/check_set_var.rs
fn main() { std::env::set_var("FOO", "bar"); }
```
- `rustc --edition=2021 ...` → **零警告零错误** ✅ (即不需要 unsafe block)
- `rustc --edition=2024 ...` → **E0133: call to unsafe function `set_var` is unsafe** ❌

项目 Cargo.toml `edition = "2021"`, 因此 writer 的 "1.79+ 标 unsafe" 陈述对**当前编译路径不准确** — `set_var` 仅在 edition 2024 语义下才需要 unsafe block。1.79 实际是引入 deprecation/future-incompat warn 的版本 (2024-06), 1.82 (2024-10) edition 2024 才正式 hard error。

**但实质 concern 真实**:
1. `setenv(3)` POSIX manpage 显式 "not thread-safe" — multi-threaded 进程里同时 setenv + getenv 可能 use-after-free (libc 内部环境表指针 race)
2. `cargo test` 默认并行执行 (rayon thread pool), 其它测试 thread 在跑 `getenv` (例: tracing / 日志库读 RUST_LOG, locale 读 LANG) 时本测试 setenv → race 条件成立
3. Rust stdlib doc 对 `set_var` 显式警告 "It is generally a bad idea to set or unset environment variables [...] in multi-threaded contexts"

所以**决策本身完全正确**, 只是技术陈述对项目 edition 2021 略夸大。`/usr/bin/env LANG=C LC_ALL=C` 包装方式:
- 仅影响子进程 env, 无全局副作用 ✅
- 不依赖 edition 升级 (forward compatible 到 2024) ✅
- 与 cargo test 并行模型 100% 兼容 ✅
- 测试自包含, 不依赖测试执行顺序 ✅

**independent verify ls -la /**:
```
$ env LANG=C LC_ALL=C ls -la / | head -3
total 48
drwxr-xr-x   1 root root   182 Mar 15 11:55 .
drwxr-xr-x   1 root root   182 Mar 15 11:55 ..
```
✅ 第一行 `total N`, 多个 `drwx` 行存在。

**接受偏离**, 推荐 follow-up 修正 module doc 措辞 (P3-1):
- 当前 "Rust 1.79+ 显式把 set_var 标 unsafe, 因为它非线程安全"
- 建议改 "Rust edition 2024 把 set_var 标 unsafe (1.82+, 2024-10), 即便在 edition 2021 项目里, set_var 在 cargo test 并行 thread 下也存在 setenv/getenv race (POSIX setenv non-thread-safe)"

### [P0-5] 测试稳定性独立复验 ✅

**5s timeout 是否够 CI 慢机**:
- 实测 `cargo test --test ls_la_e2e` 完成 0.01s, 5000× 余量
- ls -la / 在 Linux 慢机 (旧 ARM / qemu) 测试场景里 100ms 量级
- 派单 "3-5s 留余量" 写明, writer 取上限 5s, 合理

**feed_until_child_exits 退出逻辑覆盖度** (line 174-199):
- `Ok(0)` → break (EOF, Linux 罕见但合规) ✅
- `Ok(n)` → advance + 继续 ✅
- `WouldBlock` → 检查 try_wait
  - `Some(_)` (子进程已退出) → sleep 20ms (让 kernel 缓冲齐飞) → drain_residual → break ✅
  - `None` (子进程仍在跑) → sleep 10ms → 继续 ✅
- `EIO` → break (Linux PTY slave 关闭典型) ✅
- 其它 Err → panic ✅
- deadline → panic (排障留 grid 现场) ✅

**关键路径**: ls 退出后 PTY master 缓冲可能仍有未读字节 (如最后几个 byte stuck 在 OS pipe 缓冲, child 已 exit 但 read 还没拿全)。20ms sleep + drain_residual **就是为这个窗口设计**, 派单 "已知陷阱" 提示完整复用。

**drain_residual no-timeout 假设独立 verify** (line 204-212):
- Linux PTY master buffer 默认上限 ~ 64 KB (kernel `pty_buffers` slab)
- ls -la / 输出量: ~25 行 × ~80 字符 + 少量 SGR 转义 ≈ 2-3 KB (远低于 buffer 上限)
- drain 循环退出条件: `Ok(0) | Err(any)` — `Err(WouldBlock)` 在子进程退出后 buffer 排空时立即触发, drain **不可能死循环**
- worst case: 如果有 64KB 数据待 drain + read syscall ~1μs/4KB → drain 完成 ~16μs, 完全 negligible

**接受 drain_residual 无显式 deadline 设计**, 是基于 PTY buffer 上限 + Err catch-all 的合理简化。

### [P0-6] GNU vs BSD ls 假设独立 verify ✅

quill 目标是 Linux only (CLAUDE.md 明示 "Wayland-only"), 测试假设 GNU coreutils ls (Arch / Debian / Ubuntu / Fedora 等主流 distro 默认):

### [P2-1] Helper 抽象合理性 ✅

`feed_until_child_exits` / `drain_residual` / `collect_all_grid_lines` 三个 helper 抽出, 而非 inline 在两测试里:

| 选项 | 评价 |
|---|---|
| A. 全 inline (每测试自己 loop) | ❌ 重复 ~30 行 read loop, 修 timeout / try_wait 逻辑要改两处 |
| B. 抽 1 个大 fn (do_ls_test) | ❌ 把 grid size + assertion 也包了, 失去测试独立性 |
| **C. 抽 3 个小 helper (writer 选)** | ✅ feed_loop 跨测试共享, drain 是 feed 内部, collect 跨测试; 每测试只负责自己的 spawn + assert |

writer 选项最优。三 helper 都是**纯 IO + 数据移动**, 不掺测试断言, 各自 ≤ 30 行, 单一职责清晰。

### [P2-2] 与 pty_echo / pty_to_term 风格一致性 ✅

逐条比对:

| 模式 | pty_echo (T-0206) | pty_to_term (T-0301) | ls_la_e2e (T-0307) |
|---|---|---|---|
| 循环 read + WouldBlock + sleep 10ms | ✅ | ✅ | ✅ (sleep 10ms 不变) |
| EIO 视作 EOF | ✅ | ✅ | ✅ |
| timeout panic + 排障字串 | ✅ (2s) | ✅ (2s) | ✅ (5s, 输出量大需更长) |
| 关键 grep 提前 break | windows(5) "hello" | line_text(0).contains("hello") | scrollback_size(已读全部内容到 try_wait Some) |
| try_wait 兜底 | ❌ | ❌ | ✅ (新增, ls 输出量大需要) |

**新增 try_wait 兜底是 T-0307 必需**: ls 输出量比 echo 大 ~500×, 子进程在最后写完 stdout 到 exit 之间有 race window, 不靠 try_wait 兜底容易 timeout-then-fail。`tests/pty_echo.rs::drop_cleans_up_long_running_child` 用 pgrep + drop 验证退出, 思路类似但路径不同。

风格一脉相承, 加 try_wait 兜底是合理增强。

### [P2-3] ls_la_smaller_grid_truncates_lines (40×24 wrap) 加项合理 ✅

派单 #B 写 "(可选) cols=40 时, ls 输出长行被 alacritty 截到 40 cols, 验证截断正确"。

writer 实装做了这个可选项, 加了 `viewport.iter().all(|l| l.chars().count() == 40)` 严格断言:

**严格 == 40 断言**: line_text 实现 (`(0..cols).map(|c| row[Column(c)].c).collect()`) 给每个 cell 的 `.c` (Cell.c 是 char), 不 trim 末尾空白。所以即便 ls 输出某行只占 30 字符, line_text 仍 pad 到 40 (空白 cell 默认 ' '). **`chars().count() == 40` 严格断言 line_text 不丢字符 / 不多 pad**, 防 alacritty 升级时 line_text 实现回归。**加分**。
