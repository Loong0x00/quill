### [P2-1] 单测覆盖 ✅
3 个 integration test 通过 `bash -c "echo cols=$(tput cols) lines=$(tput lines)"` 端到端验 kernel winsize 真落到 slave:
1. `spawned_size_reflects_in_tput`: spawn 80x24 → 断言 `cols=80 lines=24`
2. `resize_before_tput_changes_reported_size`: spawn → resize(120,40) → 断言 `cols=120 lines=40`
3. **`resize_arg_order_cols_rows_not_swapped`**: spawn → resize(150,50) → 断言 `cols=150 lines=50` **AND NOT** `cols=50` **AND NOT** `lines=150` — 150x50 不对称能区分位置错乱, 写码主动对应 Implementation notes 第一条"cols/rows 别传反"警告加防回归, 加分

## P3 建议 (不阻塞)

### [P3-1] resize 后 tput 能看到新尺寸的 timing 假设
test 2/3 依赖 "本进程 resize ioctl 完成 → bash 启动到 tput" 时间差, 实测 49 tests 全过稳定。极端快 bash + 极端慢 ioctl 理论上可翻车。

解决路径 (若未来慢机 flaky): resize 前后 `sleep(10ms)` 给 ioctl 稳时间, 或 script 改 `sleep 0.05; echo cols=...`。

### [P3-2] pixel_width/height 填法 Phase 4 再决定
docstring 伏笔 "Phase 4 HiDPI 接入时可能填 cell 像素尺寸", 实际 TIOCSWINSZ 的 ws_xpixel/ypixel 历史上是 CRT 终端像素, 现代终端多填 0 或 cell*像素, 具体看 alacritty_terminal 约定。

归档, 非技术债 (Phase 4 实装时决定)。
