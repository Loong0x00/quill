## P0: 无阻塞

### [P0-1] propagate_resize_if_dirty 抽到 drive_wayland 是否合理 ✅
**这是 T-0306 最关键架构判断点**, 我独立分析过再下结论, 不被 writer 给的合理化叙事直接说服。

**事实矩阵**:
- `WindowHandler::configure` trait fn 签名: `fn configure(&mut self, _conn: &Connection, _qh: &QueueHandle<Self>, _window: &Window, configure: WindowConfigure, _serial: u32)` — 第 0 参 `&mut Self` 即 `&mut State`
- 当前 `struct State` 7 字段: registry_state / output_state / renderer / window / conn / core / pty — **零 term 字段**
- `term: Option<TermState>` 在 `LoopData` (run_window 局部 struct), 不在 State

**writer 的两个选择空间复验**:

| 选项 | 优 | 劣 | 评价 |
|---|---|---|---|
| A. 把 term 挪进 State, configure 直调 | 单回调内闭环 | **违反 handoff §3 显式不变式** "term 放 LoopData 而非 State 的理由:wayland Dispatch 回调不需要 term, 放 State 里会污染 Dispatch mental model" | ❌ 违反既定架构 |
| B. 抽 propagate_resize_if_dirty(LoopData), drive_wayland 接通 | 消费者单一 (INV-006); split borrow 从 LoopData 字段做; configure callback 保持 wayland-only | dirty 与消费有"窗口期"(几行代码) | ✅ 接受 |

**handoff §3 原话** (引): "Term 放 LoopData 不放 State — wayland Dispatch trait 不需要 term, State 干净专注 wayland callback"。这条不是 writer 现做现说的合理化, 是项目既定不变式。**选项 A 会反转 T-0108 的架构决策, 影响远超本 ticket**。

**语义等价性独立复验** (是不是真的"无延迟"):
1. wayland fd readable → calloop 调 `drive_wayland`
2. `dispatch_pending` 期间 `WindowHandler::configure` 跑 `handle_event` 置 `core.resize_dirty`
3. dispatch_pending 返回后 (Step 3.5 注释), 紧接 `propagate_resize_if_dirty` 消费 dirty 同步 renderer/term/pty
4. flush 推 ack_configure 给 compositor
5. 下一次 idle callback (event_loop.run 的第三参数) 看到 term.is_dirty (resize 副作用) → draw_cells 用新 cols/rows 重画

第 2 → 第 3 之间的"窗口期"只有 dispatch_pending 函数返回到 propagate_resize_if_dirty 调用之间几行代码, **同一线程同一 tick 内闭环, 不跨 ppoll 唤醒**, 无观测者可见。

**INV-006 消费者单一性增强**: T-0306 改前 `WindowHandler::configure` 的 init 路径 `} else { self.core.resize_dirty = false; }` 兼任清零;T-0306 删除该清零, 把 dirty 清零职责迁到 propagate_resize_if_dirty (唯一上游消费)。改后:
- 置位:`handle_event(Configure)` (单一)
- 清零:`propagate_resize_if_dirty` (单一)

INV-006 entry "上层 (T-0103 wgpu swapchain 重建者) 必须显式清零" — 现在"上层"具体落到 propagate_resize_if_dirty 一处。**消费者单一性强化, 不违反 INV-006**。

**接受 writer 选择**, 理由非 rubber-stamp:
1. 选项 A 会违反 handoff §3 既定架构不变式 (term 不进 State)
2. 选项 B 语义等价无延迟 (同 tick 内闭环), INV-006 消费者单一性强化
3. propagate_resize_if_dirty 抽出后 split borrow `let LoopData { state, term, .. } = &mut *data` 干净, 复用 T-0108 + T-0305 idle callback 同款套路 (handoff §3)

### [P0-4] 0 防御策略不一致 (intentional, 接受) ✅
不一致存在, 但**有架构正当性**:

| 位置 | 0 处理策略 | 理由 |
|---|---|---|
| `TermState::resize` | `cols.max(1), rows.max(1)` clamp | 终端 grid 不变式: cursor 必有 cell 落点, 1×1 仍合法状态机 |
| `Renderer::new` (既存) | `width.max(1), height.max(1)` clamp | 构造期必返 Self, 不能 Err for size 0 |
| `Renderer::resize` (新) | `width == 0 \|\| height == 0 { return; }` no-op | 1×1 surface 几乎无用, **保留上次有效 config** 等下次正常 resize 自动恢复, 比 clamp 到 1×1 后再被覆盖少一次 surface.configure 浪费 |
| `cells_from_surface_px` | `cols.max(1), rows.max(1)` clamp | 与 TermState::resize 对偶, 给 term/pty 永远是合法 grid |

**不一致是 trade-off 不是 bug**:
- 终端层 (term + pty) 必须始终在合法状态 (≥1×1) — 下游消费者读 dimensions 假设 ≥1×1
- Renderer 层 0 是 "不变更", 等真值来再变 — 等价于 "skip wasteful intermediate state"

**deeper sanity check**: `propagate_resize_if_dirty` 调 `Renderer::resize(state.core.width, state.core.height)`。`state.core.width / height` 由 `handle_event` 维护, **handle_event 已显式吞掉 0 轴 configure** (window.rs:140-142 `if w == 0 || h == 0 { return action; }`)。所以到 propagate_resize_if_dirty 时 width/height 不可能为 0。Renderer::resize 的 0 guard 是 defense-in-depth, 不是真实路径。

writer 注释 (render.rs:264-269) 显式说明该取舍, 符合 conventions §2 "why 不写 what" 且引上下游不变式。**接受**。

### [P0-5] resize_sets_dirty "相同尺寸也置 dirty" 单测合理性 ✅
独立分析"调用方一致语义" vs "冗余 spike":

**测试断言** (term/mod.rs:1432-1444):
```rust
t.clear_dirty();
t.resize(100, 30);
assert!(t.is_dirty(), "resize 后应 dirty");

t.clear_dirty();
t.resize(100, 30);  // 相同尺寸
assert!(t.is_dirty(), "resize 到相同尺寸也应置 dirty (调用方一致语义)");
```

**论点 A (writer 主张)**: 调用方一致语义。term.resize 是简单 setter, 不做"是不是真变了"过滤。"相同尺寸也置 dirty" 等价于沿袭 T-0301 advance 模式 (空切片也置 dirty)。

**论点 B (审码独立)**: 浪费一帧。但**真路径不会触发此情况** —— `propagate_resize_if_dirty` 只在 `core.resize_dirty == true` 时跑, 而 `core.resize_dirty` 由 handle_event 设定, 后者已**幂等过滤同尺寸 configure** (state_machine.rs::idempotent_same_size_configure_does_not_re_dirty 锁住, 即 INV-006 不变式)。所以真路径不会出现"相同尺寸也调 term.resize"。

**结论**: 测试锁住的是 **TermState::resize 自身的简单 setter 语义** (不做内部去重判断), 而**幂等过滤是上游 handle_event 的责任**。两条职责分离干净, 各自有单测覆盖:
- `TermState::resize` 单测: 调即 dirty (本测试)
- `handle_event(Configure)` 单测: 同尺寸不 re_dirty (state_machine.rs 既存)

这是 conventions §3 抽状态机模式的正确应用 — 不变式分层, 单测各盖一层。**接受**。

writer 注释明确 "alacritty Term::resize 在尺寸不变时内部 early return (debug log "Term::resize dimensions unchanged"), 但我们仍置 dirty 让调用方不必区分 '真 resize / 假 resize'" — 决策追溯链清晰。

### [P0-7] LoopData split borrow 正确性 ✅
独立追迹 `propagate_resize_if_dirty` (window.rs:437-487) 借用关系:

```rust
fn propagate_resize_if_dirty(data: &mut LoopData) {
    if !data.state.core.resize_dirty { return; }              // 早返 (data 未被 split)
    let LoopData { state, term, .. } = &mut *data;            // split: state: &mut State, term: &mut Option<TermState>

    let width = state.core.width;                              // 读 state.core
    let height = state.core.height;                            // 读 state.core
    let (cols, rows) = cells_from_surface_px(width, height);   // 纯 fn, 无借用

    if let Some(r) = state.renderer.as_mut() { r.resize(width, height); }
    //              ^^^^^^^^^^^^^^^^^^^^^^^^ mut state.renderer (释放在 if-let 出口)

    if let Some(t) = term.as_mut() { t.resize(cols, rows); }
    //              ^^^^^^^^^^^^^^ mut term (与 state 字段不冲突)

    if let Some(p) = state.pty.as_ref() {
        if let Err(err) = p.resize(cols as u16, rows as u16) {
        //                 ^^^^^^^ p.resize 是 &self (PtyHandle::resize 内部 ioctl, T-0204)
            tracing::warn!(...);
        }
    }

    state.core.resize_dirty = false;                           // mut state.core (前序借用全释放)
    tracing::debug!(...);
}
```

**正确性**:
- LoopData 字段级 split borrow (state, term 是不同字段) — Rust NLL ✅
- State 字段级顺序借用 (core / renderer / pty 各取各的, 不并行) — Rust NLL ✅
- pty.resize(&self) 与 pty_read_tick (&mut pty) 不在同一调用栈, 无竞争 ✅ (handoff §3 提示验证)
- term.resize(&mut term) 与 idle callback 的 term.cells_iter() 不在同一调用栈 ✅
- 编译通过即对 ✅ (cargo build 0.07s)

**调用顺序 (renderer.resize → term.resize → pty.resize)**:
- 顺序无强约束 (三资源独立)
- writer 注释 (window.rs:418-426) 明确 "顺序无强约束, 但顺序固定便于排障"
- 1. renderer.resize 先: surface 已就绪, 下次 draw_cells 自动用新 NDC 分母
- 2. term.resize: alacritty grid + cursor clamp + dirty 置位 (触发 idle callback 重画)
- 3. pty.resize: TIOCSWINSZ ioctl (失败容错, 仅 warn)

**容错设计正确**: pty.resize 失败 (rare, ioctl EBADF 或 fd 提前关) 仅 tracing::warn 不 panic / 不退出 — UI 仍可用 (term 已 resize, 渲染正常), 仅 shell 不知新 winsize, 退化非致命。

## P2: API 设计

### [P2-1] TermState::resize 与 alacritty Term::resize signature 关系 ✅
- alacritty: `Term::resize<S: Dimensions>(&mut self, size: S)` — 接 Dimensions trait 实现者
- quill: `TermState::resize(&mut self, cols: usize, rows: usize)` — 接两 usize, 内部包成 TermSize

**取舍**:
- 不暴露 TermSize (它是 alacritty Dimensions trait wrapper, 暴露 = 类型隔离漏)
- 接两 usize 比接 (cols, rows) tuple 易读 (位置参数语义清楚, IDE 帮你 hint)
- 与 TermState::new(cols: u16, rows: u16) signature 对偶 (但 new 用 u16 是兼容 alacritty `Term::new` 旧接口, resize 用 usize 因为 alacritty Term::resize 内部用 usize) — 类型不同是历史包袱, 不阻塞

类型隔离 SOP 第五次应用沿袭 T-0302..T-0305 标准。

### [P2-2] Renderer::resize 与 wgpu Surface::configure 关系 ✅
- wgpu: `Surface::configure(&self, device, config)` — 全配置注入 (含 format, present_mode 等)
- quill `Renderer::resize(&mut self, width: u32, height: u32)` — 仅尺寸更新, 其余 self.config 字段保持

**关系清晰**: Renderer 内部维护 self.config 全部状态 (含 format / alpha_mode / present_mode 等); resize 只暴露尺寸接口给上层, 内部更新 width/height + 调 surface.configure。Renderer::new 走完整构造路径; Renderer::resize 走差量更新路径 — 与 Outdated/Lost 重配 (走 self.config 不变重建) 形成三档:

| 触发 | self.config 是否更新 | 调用 |
|---|---|---|
| Renderer::new | 全新 | 初始化 |
| Renderer::resize (T-0306) | 仅 width/height | Wayland configure 尺寸变化 |
| Outdated/Lost (既存) | 不变 | 透明重建, swapchain 错误恢复 |

API 内聚, 各档职责清晰。

### [P2-3] 调用顺序 renderer → term → pty 合理 ✅
独立分析三档顺序方案:

| 方案 | 优 | 劣 |
|---|---|---|
| pty 先 (notify shell ASAP) | shell 早一步知道新尺寸, 减少误重排 | 若 pty.resize 失败 panic UI 没更新 (writer 是 warn 不 panic, 此劣不成立但仍可改) |
| term 先 (内部状态先就位) | term 内部 cursor clamp 完成后再做其他 | 顺序无观察者 |
| **renderer 先** (writer 选, surface 先就位) | 下次 draw_cells (idle callback) 立刻用新 NDC 分母 | 若 renderer.resize 内部 panic (理论上不会) UI 卡住但 term/pty 也未更新 — fail-fast |

writer 选择合理 (1: renderer surface ready → 2: term grid + dirty trigger 重画 → 3: pty.resize 异步 SIGWINCH bash)。三步均独立无强约束, 顺序固定便于排障 (writer 注释明示)。

### [P2-4] cells_from_surface_px 抽纯函数模式 ✅ (conventions §3 应用)
独立审视: 为啥不直接 inline 在 propagate_resize_if_dirty 里?

**抽出理由**:
- conventions §3 鼓励"复杂决策抽纯函数 + headless 单测覆盖" — resize 数学决策 (整数除 + max(1) clamp) 不复杂但是边界 case 多, 4 个单测覆盖比黑盒经过 propagate_resize_if_dirty 间接测试更严
- 与 `pty_readable_action` (T-0205) / `handle_event` (T-0107) 同款套路 — 副作用与决策分离
- 测试覆盖: default 锁住默认值, grows_with_surface 验拖大, clamps_zero_to_one 验防 0, truncates_partial_cells 验整数除截断 — 4 case 各打一类

写码 comment (window.rs:404-406) 明确 "抽成纯 fn (无副作用、不碰 LoopData) 是 conventions §3 '复杂决策抽纯函数 + 单测' 套路的复用" — 决策追溯引用准确。

### [P2-5] propagate_resize_if_dirty 调用位置 (drive_wayland Step 3.5) ✅
独立审视调用位置选择:

| 位置 | 优 | 劣 |
|---|---|---|
| dispatch_pending 之前 | n/a | configure 还没跑, dirty 没置, 早调白跑 |
| **dispatch_pending 之后, flush 之前** (writer 选) | dirty 刚置位立即消费, ack_configure 还没推 → flush 跟随 | 与 core.exit 检查的顺序: writer 先做 exit→stop, 再 propagate, 再 flush |
| flush 之后 | ack_configure 已推, 但 surface 尺寸滞后一帧 | 视觉延迟 |

**writer 选项最优**。一个 micro-concern: drive_wayland Step 3 检查 `core.exit → loop_signal.stop()` 在 propagate 之前。若 init_renderer_and_draw 失败导致 exit=true, propagate 仍会跑一遍 (renderer=None 跳过, term/pty 同尺寸 resize 是无副作用)。**无害**, 仅 micro 多余 (P3-2 注)。
