## P0 全部通过

### [P0-1] INV-008 / INV-009 新增 ✅
- INV-008: `PtyHandle` 字段声明顺序 `reader → master → child` = Rust 正向 drop 序
  - reader 先 drop 释放 `try_clone_reader` 的 dup fd
  - master drop 关主 fd → slave 端读到 EOF 并/或收到 SIGHUP
  - child 最后 drop (不阻塞 wait, 未 reap 子进程由进程退出 / T-0205 兜底)
- INV-009: master fd 在 `spawn_program` 返回前必须 `O_NONBLOCK`
  - 防 T-0203 读字节阻塞整个事件循环 (Ghostty starve 坑反面)
- `docs/invariants.md` 仅**追加**, 未改动 INV-001..007 任一条

### [P0-2] unsafe SAFETY 完整 ✅
`set_nonblocking` 的 unsafe 块覆盖 4 条前置条件:
1. fd 来自 `openpty` 刚建 master 端, 进程内活跃, 构造期单线程无别 clone
2. `libc::fcntl` 对已关 fd 返回 EBADF (非 UB)
3. `F_GETFL` / `F_SETFL` 只读写 open-file-description 级别 flags, 不释资源不改所有权 (不与 INV-008 drop 顺序相作用)
4. 两次 fcntl 都按 `< 0` 判错, 转 `io::Error::last_os_error`, 无静默吞错

### [P0-3] 字段顺序 = drop 序 ✅
```rust
pub struct PtyHandle {
    reader: Box<dyn Read + Send>,
    master: Box<dyn MasterPty + Send>,
    child: Box<dyn Child + Send + Sync>,
}
```
无自定义 `Drop` impl — 纯靠声明顺序。设计简洁, 避免 Drop impl 的 panic/绕回复杂性。

### [P0-4] slave spawn 时序 ✅
`pair.slave.spawn_command(cmd)` 在 `drop(pair.slave)` 前, 符合 portable-pty 文档要求 (否则 master 读不到 EOF)。T-0201 的 Implementation notes 已复述这个坑。

### [P0-5] 异常路径不漏 O_NONBLOCK ✅
只有一条 spawn 路径, `set_nonblocking` 在 `PtyHandle` 组装前调用。spawn 失败 (openpty / spawn_command / as_raw_fd / set_nonblocking 任一失败) 返回 `?`, fd 随 `pair.master` drop 关闭, 不泄露。
