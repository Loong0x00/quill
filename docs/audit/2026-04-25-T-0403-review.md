### [P0-4] draw_glyphs → draw_frame 偏离评估 (writer 主动告知 #2) ✅ 接受

**派单 In #B 写**: `pub fn draw_glyphs(&mut self, glyphs: &[(ShapedGlyph, CellPos, fg_color)], cell_w_px: f32, cell_h_px: f32) -> Result<()>`

**派单 In #C 写**: "idle callback 在 draw_cells 之后调:1. text_system.shape_line(line_text) for each viewport row 2. shape 完拿 glyphs → renderer.draw_glyphs(...)"

**实装**: `pub fn draw_frame(&mut self, text_system: &mut TextSystem, cells: &[CellRef], cols: usize, rows: usize, row_texts: &[String]) -> Result<()>` 单一入口 clear + cells + glyphs + present 同帧。

**writer 论点**: wgpu Surface `get_current_texture` / `frame.present` 必须同帧成对, 字面 spec 实施需要两次 acquire/present 视觉上撕裂 (cells 帧 N 与 glyphs 帧 N+1, 用户拖窗口快时观察到字飘忽)。

**独立审视 wgpu Surface 语义**:

wgpu 0.29 `Surface` 工作流:
1. `surface.get_current_texture()` → SurfaceTexture (含 frame view)
2. encoder.begin_render_pass(...) → 写入 frame
3. queue.submit(encoder.finish())
4. `frame.present()` → 把 frame 显示到屏幕, 释放 swapchain slot

**关键约束**: 一个 SurfaceTexture 对应**一个 swapchain slot**, present 后该 slot 归还 swapchain, 下次 `get_current_texture` 拿新 slot (新帧)。两次 acquire/present 必然是两帧 (swapchain 至少 2 slot, 可能 3-4 slot 双/三缓冲)。

**字面 spec 实施分析**:
- draw_cells (T-0305 既有): 内部 acquire frame_N + render cells + present
- draw_glyphs (派单字面): 内部 acquire frame_N+1 + render glyphs (但缺 cell 色块) + present
- 视觉结果: frame_N 显示色块无字, frame_N+1 显示字无色块 — **物理上撕裂**

**Goal-driven 评估**:
- 派单 Goal 原文: "屏幕上 bash prompt **真显示成 ASCII 字符** (深蓝背景, 浅灰字, 单色无 anti-aliasing 灰阶 OK)"
- 派单 Acceptance 第 6 项: "**手测**: cargo run --release 起窗口 → 屏幕显示 bash prompt **真字符** (不是色块)"
- Goal 隐含 cells (深蓝背景 / 色块) 与 glyphs (字符) **同帧 present**, 否则视觉上"看到字"是失败的

**writer 选择 draw_frame 单一入口** 评估:
- ✅ 单 RenderPass clear + cells (BlendState::REPLACE) + glyphs (BlendState::ALPHA_BLENDING) 同帧 ✅
- ✅ acquire frame 一次, render 两 pipeline, submit + present 一次 — wgpu 标准 multi-pipeline 单 pass 模式
- ✅ Goal 物理满足 (cells 与 glyphs 同帧 present, 无撕裂)
- ✅ draw_cells 保留作 fallback (text_system None 时降级 Phase 3 色块路径)
- ⚠️ 偏离派单字面 In #B + #C 措辞, 但 conventions §5 "派单 vs 实装对照, 不静默偏离" 已写在 commit body
- ✅ 沿袭 T-0305 fg vs bg + T-0306 Renderer::resize 偏离主动告知模式 (审码 T-0306 P0-2 范例)

**draw_cells fallback 保留是否合理**:
- text_system 启动期 init 失败 (CI 无 monospace 字体) → text_system = None → idle callback 走 draw_cells 色块路径
- writer 注释 (window.rs:734-744) 显式 graceful degradation 路径 + tracing::warn 记 init 失败
- ⚠️ **P3 微观察**: 失败路径降级是 graceful 还是吞错? Phase 4 Goal "看见字", 字体加载失败应明确报错 (panic 或 exit) 而非降级 (色块)。但用户机 noto-cjk + dejavu 装齐, init 失败极罕见 (CI 退化路径 graceful 接受)。**接受 graceful degradation**, 现实路径不会触发。

**判**: ✅ **接受偏离** — Goal binding > Scope wording 是 T-0305 P0-6 norm, wgpu Surface 物理约束让派单字面"两次 draw"不可行, draw_frame 单一入口是 Goal-driven correction 的正确实施。
