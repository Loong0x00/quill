### [P0-6] 回调 drain 正确 ✅
drain 逻辑:
- `Ok(n>0)` continue
- `Ok(0)` EOF → return Continue (退出留给 T-0205)
- `Err(AGAIN)` → return Continue (无更多数据)
- `Err(INTR)` → continue
- 其它 Errno → 抛给 calloop error path

O_NONBLOCK 由 INV-009 保证 drain 必然在 AGAIN 退出, 不 busy loop, 不 panic。`trace!("pty readable")` 在 drain 之前调一次, 不在 loop 内, trace 量 ≈ 数据到达次数。

## P3 建议

### [P3-1] tests/pty_calloop_smoke.rs 的 `drop(handle)` 违反 BorrowedFd 合约
两测试末尾显式 `drop(handle);` 时 event_loop 仍栈上, Generic source 持 `BorrowedFd<'static>` = handle.master_fd。早 drop handle 在合约上等于 "borrowed outlives owner"。实际不 UB (calloop Generic<BorrowedFd> 的 Drop 不对 fd 做 syscall), 但合约上不对。

**Lead 要求: 合并前修正**。删 `drop(handle);` 那行, 让反向 drop 自然走 (event_loop 先 drop → handle 后 drop)。

### [P3-2] pump_once BorrowedFd SAFETY 注释详略 (不阻塞)
建议展开到 T-0201 的 4 点风格: (1) fd 来源 (2) fd 活性 (3) 借用生命周期 (4) syscall 副作用。**归档为风格偏好, 不阻塞合并**。

### [P3-3] run_window BorrowedFd<'static> SAFETY 可更严谨 (不阻塞)
建议明写 "pty_core 在 state 之后声明, Rust 反向 drop 保证 pty_core EventLoop/Generic source drop 先于 state.pty PtyHandle drop, 因此 BorrowedFd 伪造的 'static 实际被 drop-order 约束在 state.pty 寿命内"。**归档, 不阻塞**。

### [P3-4] SAFETY 注释 "unwrap" 字样误导
`src/wl/window.rs:272,408` SAFETY 注释写 "pty_fd 来自 state.pty.as_ref().unwrap().raw_fd()", 但代码实际 `let pty_fd = pty.raw_fd();` (spawn 返回值绑定, 不经 state.pty, 无 unwrap)。注释与代码不符 + "unwrap" 字样容易误导后人以为真有 unwrap。

**Lead 要求: 合并前修正**。改成 "pty_fd 来自 spawn_shell 返回值的 pty.raw_fd() 缓存, state.pty 持有 PtyHandle 到 run_window 结束"。

### [P3-5] 测试闭包 "SAFETY 提醒" 字样 (不阻塞)
tests/pty_calloop_smoke.rs:38-44 回调里 "// SAFETY 提醒:..." 是实现选择说明, 该闭包本身非 unsafe 块, 稀释 SAFETY 关键字。**归档, 不阻塞**。
