### [P0-1] INV-010 strict reading: pub(crate) accessor 是否算"暴露" ✅✅

**这是 T-0401 最高 governance 风险点**, writer 主动告知 #1, 我独立判断不被 writer 自圆其说说服。

**事实**: writer 实装中曾有 `pub(crate) fn swash_cache_mut(&mut self) -> &mut cosmic_text::SwashCache` accessor (供 T-0403 光栅化 ticket 直接拿 SwashCache 用), 自审认为 INV-010 strict reading 下 `pub(crate)` 也算"暴露", 删除该 accessor。

**INV-010 文字关键句** (docs/invariants.md:213):
> 凡是上游 crate 暴露的 enum / struct ... 一律**不得**作为本 crate `pub` 类型 ... 直接暴露

**INV-010 验证段** (docs/invariants.md:232-236):
> - `grep -nE 'pub use (alacritty|wgpu|wayland|cosmic)' src/` 应零命中
> - `grep -nE 'impl From<(alacritty|wgpu|wayland|cosmic)' src/` 应零命中

**两种合理解读**:

| 解读 | INV-010 文字字面 | 治理价值 (cascade refactor 防御) |
|---|---|---|
| **A. 仅 `pub` 算暴露** (`pub(crate)` 不算) | 严格按字面 — INV-010 写 "`pub` 类型" 不写 "`pub(crate)` 类型" | 弱 — `pub(crate) cosmic_text::SwashCache` 让 T-0403 光栅化 ticket / 未来 atlas ticket 直接持 cosmic 类型, 升级 cosmic-text 主版本仍 cascade 改跨模块调用点 |
| **B. `pub(crate)` 也算暴露** (writer 自判 + 删除) | 字面外延 — `pub(crate)` 是 crate 级暴露, 跨模块调用点都触类型路径 | 强 — 单一改动点真落实, 升级 cosmic-text 只动 src/text/mod.rs |

**我的独立判断**: **B 是 quill 项目应该坚持的 reading**, 理由:

1. **INV-010 设计目的明示** (entry "为啥" 段, docs/invariants.md:220-225): "换 VT 库 / wgpu 主版本升级 / wayland-protocols 大版本时**必须 cascade 改下游所有调用点**"。`pub(crate)` accessor 让其他模块 (例 src/wl/render.rs) 直接持 `&mut cosmic_text::SwashCache` → src/wl/render.rs 也变成 cascade 改动点 — 设计目的破口

2. **CellPos / CursorShape / Color 三类型先例**: T-0302..T-0305 五次类型隔离 SOP 应用时, `from_alacritty` 全部是**模块私有 inherent fn** (无 `pub` / 无 `pub(crate)`), 没有任何"crate 级暴露"先例

3. **conventions §5 心法原话** (docs/conventions.md:222): "`impl OurType { fn from_alacritty(p: alacritty::Point) -> Self { ... } }` **模块私有 inherent fn**, 下游构造不出来" — "下游" 包含同 crate 其他模块, 不限 crate 外用户

4. **替代设计可行**: T-0403 光栅化不需要直接持 SwashCache, 在 TextSystem 上加 `pub fn raster_glyph(&mut self, ...) -> RasterizedBitmap` (返回 quill 自定义 bitmap struct) 即可。"单一改动点" 真意义是渲染层调用 raster_glyph 而非操控 swash_cache

**writer 的删除决策正确** ✅。**INV-010 strict reading 是 quill 项目应该坚持的, 不是 over-engineering** — 是项目治理价值的真正落实, 6 单零违规历史 (T-0302..T-0306 + T-0399) 累积到位。

### [P0-4] ShapedGlyph 字段语义 ✅

**writer 主动告知 #2, #3, #4** — 我独立 verify cosmic-text 0.12.1 源码 (`/home/user/.cargo/registry/src/index.crates.io-1949cf8c6b5b557f/cosmic-text-0.12.1/`) 而不仅依赖文档描述:

### [P0-5] swash_cache #[allow(dead_code)] 取舍 (writer 主动告知 #5) ✅

**两个设计选项** (我独立分析):

| 方案 | 优 | 劣 |
|---|---|---|
| **A. TextSystem 持 swash_cache 字段 + #[allow(dead_code)]** (writer 选) | 资源生命周期一处定义, 后续 ticket 只加 method 不改字段; 沿袭 PtyHandle (5 method 共 1 struct) / Renderer (10 字段共 1 struct) 项目模式 | 字段当前不消化, 编译期需 `#[allow(dead_code)]` 显式 silence |
| **B. 拆 TextSystem (FontSystem only) + TextRaster (FontSystem + SwashCache)** | 无 dead_code allow; 单 struct 单职责 | T-0403 时改 TextSystem 字段或拆新 struct = 改 owned 资源链, 接班 reviewer 需重读资源生命周期; 与 PtyHandle / Renderer "一资源一 struct" 模式不一致 |

**我接受 A** ✅, 理由:

1. **派单 scope 显式列字段** (`pub struct TextSystem { font_system: cosmic_text::FontSystem, swash_cache: cosmic_text::SwashCache }`) — writer 严格按派单, 不静默偏离 (conventions §5)

2. **`#[allow(dead_code)]` 注释引派单 + T-0403 接入路径** (src/text/mod.rs:42-49) traceability 完整, 后续 ticket grep 接入点直达

3. **项目"一 struct 多 method"模式一致**: `PtyHandle` 持 reader/master/master_fd/child 4 字段, 5 method 复用; `Renderer` 持 10 字段, 多 method 复用; T-0401 引入 TextSystem 持 2 字段, T-0401 用 1 + T-0403 接入 1 — 一致

4. **拆两 struct 的真实成本**: 不止改字段, 还需 ctor 关系 (TextRaster::new 需引用 TextSystem 的 FontSystem 还是各自持一份?), 引入 lifetime / Arc 复杂度。本 ticket 不应做这种 architectural 决策

5. **dead_code allow 的负面影响有限**: 字段是 owned `cosmic_text::SwashCache`, 不影响 INV-010 grep (已 verify 零命中); 编译期 silence 是局部代价, 不污染 src/wl 或公共 API

**T-0403 ticket 设计建议** (给 Phase 4 后续 reviewer 留): T-0403 在 `impl TextSystem` 加 `pub fn raster_glyph(&mut self, gid: u16, font_size: f32) -> RasterizedBitmap` — 内部用 `self.swash_cache.get_image(...)` + 转 quill 自定义 `RasterizedBitmap` struct, 不暴露 cosmic_text::SwashImage。届时 `#[allow(dead_code)]` 可删除。

**shape_chinese_zhong_returns_glyph** (src/text/mod.rs:231-248):

let g = ts.shape_one_char('中');

writer 注释 "接受两种结果(有 glyph 且 advance > 0 / tofu glyph),只要不 panic 即合规" — 但实际 `assert!(g.is_some())` **要求 Some**, 不接受 None。

**结论**: `assert!(g.is_some())` 实际上是合理强假设, CI 没 CJK 字体退化到 .notdef 仍是 Some(ShapedGlyph) — writer 注释 "接受两种结果" 在实际行为上会触发 `assert!(g.is_some())` 通过 (退化路径仍 Some)。**注释表述与代码 strict 度有微张力但行为正确** — P3 微观察。

**shape_unknown_codepoint_returns_some_or_none_no_panic** (src/text/mod.rs:253-273):
```rust
let g = ts.shape_one_char('\u{E0000}');
if let Some(sg) = g {
    assert!(sg.x_advance.is_finite(), "...");
    assert!(sg.y_advance.is_finite(), "...");

```

- ✅ 不 assert `g.is_some()` (是真宽松)
- ✅ `is_finite()` NaN 防御 — 派单 "已知陷阱 5" 提到 NaN 防 NDC 换算污染
- ✅ U+E0000 是 Unicode tag space 私用区, 通常字体不含 — 测试目标是 cosmic-text 不 panic

**NaN 触发场景独立审视**: cosmic-text shape pipeline 中 `glyph_font_size * glyph.x_advance` (shape.rs:1459) 都是 f32 乘法, 输入有限则输出有限。NaN 来源仅可能是:
1. `font_size` NaN (本 fn 用 `Metrics::new(14.0, 20.0)` 都是常数, 不 NaN)
2. `glyph.x_advance` NaN (来自 rustybuzz / harfbuzz_rs 的 shape 输出 — 极罕见)
3. `justification_expansion` NaN (Align::Justified 路径, 本 fn 未触发)

**`is_finite()` 是 defense-in-depth, 触发概率近零, 但加价值是单测固化"glyph advance 必为 finite" 不变式, Phase 4 后续 ticket 改 shape pipeline 时拦回归**。接受。
