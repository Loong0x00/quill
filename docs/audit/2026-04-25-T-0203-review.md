### [P3-2] Ok(0) / EIO 的 busy-warn 风险 — **T-0205 必须接手**
Level-triggered + master EOF → fd 永久 readable。T-0205 实装前若 shell 异常退出, pump_once 每轮 dispatch 一次 callback → warn 一行 → 立刻又一轮, 估计每秒数千行 busy-warn。

**Lead 拍板 T-0205 路径: 方案 3 Arc<AtomicBool>**。理由:
- T-0104 已有 `Arc<AtomicBool> should_exit` 模式, 复用零新架构
- 不等 T-0105 refactor 阻塞 T-0205
- tuple Data (`(&mut PtyHandle, &AtomicBool)`) 难看
- callback 里 clone 一份 Arc, set true, run_main_loop 顶部轮询自然退出

已登记 TD-012 + T-0205 ticket Implementation notes 更新。

### [P3-3] test2 注释误导 "echo 父 shell" — **Lead 要求合并前修**
`src/pty/mod.rs:343` 注释:"可能第一次就 WouldBlock, 也可能 echo 父 shell 写出啥来 — 抢最多 1ms 多读几次"。

本测试 spawn 的是 `/bin/sleep 0.1`, 不是 echo, 不是 shell。注释混淆 spawn 目标 (类似 T-0202 P3-4 "unwrap" 字样认知 bug)。

**改为**: "spawn sleep 0.1 不写 stdout, 理论上第一次 read 立即返 WouldBlock; 10 次重试是防某些 PTY 初始化瞬间字节"。
