### [P0-2] ShapedGlyph x_offset / y_offset 命名歧义 vs cosmic-text LayoutGlyph (writer 主动告知 #2) ⚠️ P2 但非 P0

writer 主动告知 reviewer 重点关注此项, 我独立 verify cosmic-text 0.12.1 源码 (`/home/user/.cargo/registry/src/index.crates.io-1949cf8c6b5b557f/cosmic-text-0.12.1/src/layout.rs:11-57`):

**cosmic-text LayoutGlyph 真有 5 个易混淆字段**:

| 字段 | 类型 | doc | 实际语义 |
|---|---|---|---|
| `pub x: f32` | f32 | "X offset of hitbox" | 字形左上角相对 line 起点的 X 像素位置 (LTR 累积) |
| `pub y: f32` | f32 | "Y offset of hitbox" | 字形左上角 Y 像素位置 |
| `pub w: f32` | f32 | "Width of hitbox" | 字形 advance (T-0401 已映射, 注释明示非"glyph bbox 宽度") |
| `pub x_offset: f32` | f32 | "X offset in line ... useful when you are dealing with logical units" | 逻辑坐标 sub-pixel 抖动 / 字距微调; `physical()` 中 `font_size * x_offset` 加到 `x` 上 |
| `pub y_offset: f32` | f32 | 同 x_offset | sub-pixel Y 抖动 |

**`physical()` 源 verify** (cosmic-text-0.12.1/src/layout.rs:69-83):
```rust
pub fn physical(&self, offset: (f32, f32), scale: f32) -> PhysicalGlyph {
    let x_offset = self.font_size * self.x_offset;
    let y_offset = self.font_size * self.y_offset;
    let (cache_key, x, y) = CacheKey::new(
        ...
        (self.x + x_offset) * scale + offset.0,
        truncf((self.y - y_offset) * scale + offset.1),
        ...
    );
    ...
}
```

**关键论点**: `physical()` 计算最终物理坐标用 `(self.x + font_size * self.x_offset)` —— 即"完整位置 = 累积位置 (x) + 字距微调 (font_size × x_offset)"。

**writer 设计选择**:
- quill `ShapedGlyph.x_offset` ← cosmic-text `LayoutGlyph.x` (累积位置)
- quill 不映射 cosmic-text `LayoutGlyph.x_offset` (字距微调) —— Phase 4 不做 sub-pixel rendering

**两种解读**:

| 解读 | 命名清晰度 | Phase 4 范畴正确性 | Phase 5+ 扩展空间 |
|---|---|---|---|
| **A. 接受 writer 命名** (派单 spec) | 弱 — 与 cosmic-text 同名字段语义不同, 注释才能区分 | ✅ Phase 4 monospace 整数像素对齐, sub-pixel offset = 0 接受 | 若 Phase 5+ 加 sub-pixel rendering, 名字已被占用, 需重构 |
| **B. 改名 `x_pos` / `y_pos` 或 `x` / `y`** | 强 — 与 cosmic-text `.x` / `.y` 一一对应 | ✅ 同 A | 若 Phase 5+ 加 sub-pixel, `x_offset` / `y_offset` 名字保留可用 |

**我的独立判断**: **A 接受, 但作 P2 设计观察登记** —— 理由:

1. **派单 spec 明示** (tasks/T-0402-shaping-pipeline.md Scope/In #A 末段): "如果 shape_line 需要 x/y position 字段, 加 pub `x_offset` 和 `y_offset` (cosmic-text LayoutGlyph.x / y)" —— writer 严格按派单, 不静默偏离 (conventions §5)

2. **writer 注释 traceability 完整** (src/text/mod.rs:202-210 + 244-248): 命名歧义被 doc string 双层显式标注 (struct 字段 doc + from_cosmic_glyph 字段映射 doc)。"`x_offset` / `y_offset` (T-0402 加): 字形左上角相对所在行起点的像素位置 —— 来自 [`cosmic_text::LayoutGlyph`] 的 `x` / `y` 字段(cosmic-text 把这两个字段命名为 hitbox X/Y 但语义就是"glyph 位置")。**注意命名差**: cosmic-text 自己也有 `LayoutGlyph::x_offset` / `y_offset` 字段表达"逻辑坐标抖动",我们**不**用那两个 (Phase 4 不做 sub-pixel rendering)" — 已显式提醒接班 reviewer 注意。

3. **Phase 4 范畴**: 目标"2x HiDPI 整数缩放 + CJK 字形正常" (CLAUDE.md), 整数像素 monospace cell rendering 不需要 sub-pixel offset。Phase 4 cosmic-text `x_offset` / `y_offset` 在 monospace ASCII / CJK 路径上典型为 0, 跳过不影响 rendering 正确性。

4. **Phase 5+ 风险登记**: 若未来加 combining marks (e.g. é = e + ́), 重音符号位置就用 cosmic-text `x_offset`。届时 quill 命名空间冲突, 需 rename quill 字段 (例 `x_offset` → `x_pos`) + 加新字段 (`x_offset_subpixel: f32` 真映射 cosmic `.x_offset`)。**this is a P2 future-debt observation**, 不是当前 T-0402 的设计错。

5. **派单选项 spec-sanctioned**: 派单 #A "可选扩展, 跟 T-0403 协调" 表述明示这是 spec 选择, 非 writer 自专。Lead 接受派单本身就是接受这个命名。

**结论**: writer 设计 spec-sanctioned + doc 充分, **接受**。**P2 登记** "未来若加 sub-pixel rendering 需重命名" 给 Phase 5+ 派单参考。

### [P0-4] shape_until_scroll(prune=true) vs shape_one_char 的 false 不一致 (writer 主动告知 #4) ✅ 接受

**两 fn 的 prune 参数**:
- `shape_one_char` (T-0401, src/text/mod.rs:126): `shape_until_scroll(&mut self.font_system, false)`
- `shape_line` (T-0402, src/text/mod.rs:172): `shape_until_scroll(&mut self.font_system, true)`

**writer 注释论点 (line 153-157)**: "本 buffer 是单次 shape 后立即丢的 scratch, prune=true 告诉 cosmic-text 主动释放 shape buffer 中未用的 scroll-out 行存储 —— 对单行场景无差,但与未来'多行可滚动 buffer'调用模式对齐。`shape_one_char` 使用 `false` 是 T-0401 早期实现选择,本 fn 不一并改 (out-of-scope; 该 fn 是 Phase 4 临时探测 API 即将退役)."

**独立审视**:

cosmic-text 0.12.1 `Buffer::shape_until_scroll(font_system, prune)`:
- prune=true: shape 后立即清掉 scroll buffer 中已 layout 完的行
- prune=false: 保留 scroll buffer (适合后续增量更新)

**单次 shape + 立即丢 scratch buffer 场景**: 两值无观测差 —— buffer 离开 fn 范围立即 drop, prune 不 prune 都释放。所以**实际行为等价**, 选择是论"语义意图正确性"。

**writer 选 prune=true 论点合理**: 表达"本 buffer 不预期复用 / 不增量"的意图, 对未来读者有 hint 价值。

**P3 微跟进 (不阻塞)**: shape_one_char 一致化至 prune=true 应在 shape_one_char 退役 ticket (T-0403 或 T-0406 决定) 一并清理, 现在两 API 共存期不强求。**writer 选择不并改保 T-0402 scope 边界正确**, 接受 ✅。

## P2: API 设计 / 架构判断

### [P2-1] ShapedGlyph 字段顺序 / 内存对齐 ✅

T-0402 后 ShapedGlyph 5 字段:
```rust
pub struct ShapedGlyph {
    pub glyph_id: u16,    //  2 bytes
    pub x_advance: f32,   //  4 bytes
    pub y_advance: f32,   //  4 bytes
    pub x_offset: f32,    //  4 bytes
    pub y_offset: f32,    //  4 bytes
}
```

**内存布局**: 默认 #[repr(Rust)], rustc 可重排字段最小化 padding。当前布局 (无 #[repr(C)]) 让 rustc 决定; 实际最优排列是 `f32 × 4 + u16 + 2 padding` = 18 bytes 实际占 20 (4 字节对齐)。

**`#[derive(Debug, Clone, Copy, PartialEq)]`**: 5 字段全 POD-like, Copy 派生 OK (Phase 4 后续若加 owned 字段如 `Box<RasterizedBitmap>` 会破 Copy, 届时 Renderer 改按值传递为引用传递)。**保 Copy 是好选择, 渲染层 hot path 友好**。

**字段顺序合理度**: glyph_id 在首位是 OpenType 惯例 (face 内主键); advance / offset 5 字段是 layout 几何, 按 (x/y, advance/offset) 二维或 (advance/offset, x/y) 二维分组都行。当前 (advance × 2, offset × 2) 分组合理, 渲染层接 atlas blit 时用 (offset, advance) 同时取也方便 ✅。

### [P2-2] shape_line API vs shape_one_char 关系 ✅

**两 API 是否本质相同**:
```
shape_line("a") == [shape_one_char('a').unwrap()]?
```

**独立分析**:
- 不严格相等 — Metrics 不同 (17/25 vs 14/20), 数值结果差异
- shape_line 走 multi-LayoutRun + flat_map; shape_one_char 走 first-LayoutRun.first()
- shape_line 输出 (gid, advance, offset) 5 字段; shape_one_char 输出 (gid, advance) 3 字段 (x_offset/y_offset 缺)

**这是问题吗**: 不是 — shape_one_char 是 T-0401 探测 API (脚手架), shape_line 是 T-0403 主入口 (生产)。两 API 设计目的不同, 不需要严格 superset 关系。

**接班角度**: 接班写码读 src/text/mod.rs 看到两 API 应理解 "shape_one_char 即将退役, 用 shape_line"。当前注释 (shape_one_char doc line 90 "Phase 4 临时探测 API" + shape_line doc line 137 "T-0403 渲染层主入口 —— 取代 shape_one_char 单字符脚手架") 已明示路径 ✅。

**P3 微观察 (不阻塞)**: shape_one_char 可加 `#[deprecated(note = "use shape_line; will be removed in T-04XX")]` 让编译期警告。当前未加, 但 doc 明示 + Phase 4 后续 ticket scope 决定退役时机, 不强求。

### [P2-3] shape_line_mixed_cjk 测试覆盖宽松度 (writer 主动告知 #6) ⚠️ 接受但有覆盖缺口

**writer 选择不锁 advance 数值**:
```rust
fn shape_line_mixed_cjk_returns_glyphs() {
    ...
    assert_eq!(glyphs.len(), 5, ...);
    for g in glyphs.iter() {
        assert!(g.x_advance.is_finite() && g.x_advance >= 0.0);
        assert_eq!(g.y_advance, 0.0);
    }
}
```

**writer 注释论点** (line 393-398): "CJK 双宽性质 (东亚字 advance ≈ 2x ASCII) 由字体设计保证, **本测试不做 advance 数值断言** (CI 退化到 tofu 时 advance 可能 ≈ ASCII 宽, 不能用 advance 区分 CJK / ASCII)"。

**独立审视覆盖缺口**:

| 退化场景 | 当前测试是否拦? |
|---|---|
| cosmic-text shape "你好abc" 完全失败 → 0 glyph 返回 | ✅ 拦 (`assert_eq!(glyphs.len(), 5)` fail) |
| cosmic-text shape 5 个 glyph 但顺序错乱 (e.g. "cba你好") | ❌ **不拦** (本测试不锁顺序数值) |
| cosmic-text shape 5 个 glyph 但全 advance=0 (CJK fallback 完全失败) | ❌ **不拦** (`>= 0.0` 接受 0) |
| cosmic-text shape 5 个 glyph 全 advance=NaN | ✅ 拦 (`is_finite()` fail) |

**真覆盖缺口**: "5 个 glyph 但 advance=0 / 顺序错乱" 退化路径不拦 — 是设计 trade-off。

**为什么接受**:
1. **CI 物理约束**: writer 论点正确, CI 无 CJK 字体退化到 tofu 时 advance 可能 ≈ ASCII 宽, 用 advance 数值区分 CJK / ASCII 是脆弱断言
2. **5 glyph 数 + finite + non-negative 已是合理基线**: 真"完全失败"信号 (panic / wrong count / NaN) 全拦
3. **顺序错乱风险有限**: cosmic-text 0.12 LTR 单 line shape 顺序由 BiDi level + run 切分决定, 对 "你好abc" (LTR mixed) 应该单调累积。**未来 Phase 5+ 加 BiDi / RTL 时若顺序断言成关键, 应单独加测试** (例 mixed Arabic / Hebrew → 顺序物理 vs 逻辑差异), 现在 Phase 4 不需要

4. **shape_line_ascii 已锁顺序** (`x_offset` 单调递增) 提供"LTR 顺序"覆盖

**结论**: 接受 ✅。**P3 跟进观察**: Phase 5+ BiDi 接入时若加 mixed_rtl_ltr 测试, 应锁 visual order 区分 RTL run。
