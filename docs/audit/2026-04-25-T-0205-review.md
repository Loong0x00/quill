## P0 全部通过

### [P0-1] POLLHUP bug fix 复核 ✅
旧: `fds[2].revents().contains(PollFlags::IN)`
新: `fds[2].revents().intersects(PollFlags::IN | PollFlags::HUP | PollFlags::ERR)`

**POSIX poll(2) 语义**: POLLIN 只有数据才设, POLLHUP (output-only, 即便不 request 也返) 在对端 close 设, POLLERR (output-only) 错误条件设。

**Linux PTY master kernel 行为**:
- slave 关闭 + master 缓冲非空: `revents = POLLIN | POLLHUP`
- slave 关闭 + master 缓冲已空: **仅 POLLHUP, 无 POLLIN**
- 错误状态: POLLERR

旧 `contains(IN)` 在"slave 关闭 + 缓冲空"场景漏过 → callback 不跑 → EOF 测不到 → should_exit 不置位。

T-0202/T-0203 时 callback 对 EOF/EIO 只 drop/warn + Continue 不触发退出, 所以"fd 永久 ready 但读不到"的 busy loop 风险被 Level + Ok(0) 吞掉。T-0205 接入 exit 路径 → 漏侦测 POLLHUP 整条退出链断。**scope expansion 合理, 不修 T-0205 Acceptance 不过**。

`intersects(IN|HUP|ERR)` 是 POSIX 标准 poll 使用的正确用法。修复注释清晰, 未来 agent 能懂。

## P3 建议

### [P3-1] try_wait docstring 对 portable-pty signal 行为描述有误 — **Lead 要求合并前修**
src/pty/mod.rs:184 附近 docstring 说:
> "若子进程被 signal 杀死, portable-pty 的 ExitStatus::exit_code() 通常返 0, 真正 signal 信息在 ExitStatus::signal() -> Option<&str> 里"

实际 portable-pty 0.9 unix_child.rs 里 `From<std::process::ExitStatus>` 已做:
```rust
if let Some(signal) = status.signal() {
    Self::with_signal(&format!("{}", signal), 128 + signal as u32)
}
```
即 signal 杀死时 exit_code() 返回 128+signum, 正好是 ticket 要求的 "128+signal 占位" 语义。功能正确, 但 docstring 描述的 portable-pty 行为错了, 和 T-0202 "unwrap" + T-0203 "echo 父 shell" 同类认知 bug。

**改为**: "依赖 portable-pty 在 Unix 已做的 128+signum 编码 (参 portable-pty 0.9 unix_child.rs), 本函数 `status.map(|s| s.exit_code() as i32)` 透传即可得到 shell-style 退出码"

### [P3-2] pty_readable_action 参数形态 (io::Result<usize> 引用 vs 值) 偏好
现在 `fn(&io::Result<usize>) -> PtyAction` + caller 再 `if let Ok(n) = result` 分支。也可改成 `fn(io::Result<usize>) -> (PtyAction, Option<usize>)` 一次返回避免 caller 再分支, 但单测写啰嗦。**保持现状即可, 不登记 TD**。

### [P3-3] POLLHUP/POLLERR 不必入 INV
项目里 rustix poll 目前仅 pump_once 一处, 不需抽不变式。修复的注释已说清 Linux PTY master 行为。T-0105 refactor 把 wayland 也 poll 进 rustix 时参考即可。**不登记 TD**。
