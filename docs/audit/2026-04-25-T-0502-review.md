## 代码改动审视

### A. set_buffer_scale 调用 (派单 In #A)

**位置**: `src/wl/window.rs:630`

```rust
if let Err(err) = window.set_buffer_scale(crate::wl::HIDPI_SCALE) {
    tracing::warn!(
        ?err,
        scale = crate::wl::HIDPI_SCALE,
        "wl_surface.set_buffer_scale 失败 (compositor wl_surface < v3?), \
         视觉可能 double-scale; 升级 compositor 修复"
    );
}
```

- **位置正确**: 在 `window.set_app_id` / `set_min_size` 之后, **在 `window.commit()` 之前** (commit 在 line 642)。surface 已创建 (smithay `WaylandWindow` 的 wl_surface), 满足 wl 协议 "set_buffer_scale 必须在 attach buffer 之前调" 的要求 — 本项目首次 attach 由 `init_renderer_and_draw` 内的 wgpu `surface.configure` + `r.render()` 触发, 远在此 commit 之后。set_buffer_scale 是 pending state, 与同函数末尾的 `window.commit()` 一并推给 compositor (协议: scale 是 double-buffered)。
- **调用次数 = 1**: `grep -c "set_buffer_scale" src/wl/window.rs` 为 8, 但 7 处是注释 / log message 字面量, 仅 line 630 是真函数调用。verify 通过。
- **value = HIDPI_SCALE**: 走 SCTK `WaylandSurface::set_buffer_scale(scale: u32)`, SCTK 内部已做 v3 检查 + `scale as i32` cast (验 SCTK 0.19.2 src/shell/mod.rs:96-103)。派单原文写 `surface.set_buffer_scale(HIDPI_SCALE as i32)` 是低层 wl_surface API 示例; 走 SCTK 高层 API 反而**更安全** (带 v3 fallback 走 `Result<(), Unsupported>` warn 而非 panic)。完全符合派单意图。
- **错误处理**: SCTK `set_buffer_scale` 返 `Result<(), Unsupported>`, 老 compositor (wl_surface < v3) 走 warn 路径, 不 panic — 符合派单"已知陷阱: compositor 可能不发, 不能依赖, 仅 log"。
- **doc 注释**: 22 行 block 注释引 T-0404 / T-0502 + 解释 hardcode 理由 + protocol order 理由 + SCTK 内部行为。conventions §2 三铁律全守 (why 不写 what; 引 ticket; 关键不变式说清楚)。

### B. log_output_scale + verdict_for_scale 抽象 (派单 In #C + 非派单 §3 套路)

**位置**: `src/wl/window.rs::log_output_scale` (line ~1055), `verdict_for_scale` (line ~1043), `OutputScaleVerdict` enum (line ~1031)

- `OutputHandler::new_output` + `update_output` 都调 `log_output_scale(&self.output_state, &output, "<event>")`, 用 `event: &'static str` 字段区分 trace 上下文。
- `log_output_scale` 内部:
  1. `output_state.info(output)` 拿 wl_output 的 `OutputInfo` (含 scale_factor); SCTK info 可能为 None (例 new_output 在 done event 之前先触发), 用 let-else 跳过。
  2. 对比 `info.scale_factor: i32` 与 `crate::wl::HIDPI_SCALE as i32`。
  3. 走 `verdict_for_scale` 决策: Match → `tracing::debug!`, Mismatch → `tracing::warn!`。
- `verdict_for_scale(compositor_scale: i32, our_scale: i32) -> OutputScaleVerdict` 是纯逻辑 fn, 无副作用, headless 单测易覆盖。
- `OutputScaleVerdict` 是 `pub(crate)` enum, 不跨 crate 边界 (INV-010 维持)。

**conventions §3 抽状态机套路**: 与 `pty_readable_action` (T-0205) / `should_propagate_resize` / `cells_from_surface_px` (T-0306) 同款 — 决策 (纯逻辑) 与副作用 (trace) 分离, 决策 headless 单测全 variant 覆盖, 副作用走集成测试 / 手测。审码显式赞这个套路 (conventions §3 末段)。

### C. log_output_scale 路径

`OutputHandler::new_output` (line ~982) + `OutputHandler::update_output` (line ~995) 各调用一次。`output_destroyed` 不调 (派单未要求, 也无意义因 output 已不在)。覆盖 compositor scale event 的两个真实触发点 (首次发现 + 后续 hot-plug / 设置改动)。
