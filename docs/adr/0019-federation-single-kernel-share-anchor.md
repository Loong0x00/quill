# ADR-0019: 多窗口/多设备联邦 —— 机器级单例 kernel + 共享锚窗口 = home

状态: Accepted(设计定稿 2026-07-01,**未实现**;实现按文末分砖)
关联: ADR-0015(R1 共享会话模型 + LAN 联邦愿景)、ADR-0018(E′ 共享=隔离子进程)、CLAUDE.md「T6 分砖」(砖0 多 workspace / 砖2 每客户端独立 tab)

## Context

砖1b/砖2 后,`quill --share`(或运行期 toggle)让**一个**桌面窗口把它的 tab 共享给手机。两个问题浮出来:

1. **多窗口撞车(实测 bug)**:开两个 quill 窗口都点共享 → 第二个的 kernel 子进程去绑 `0.0.0.0:7878` 撞车退出。已用 ADR-0018 的"启动失败→标题栏 error 态"兜底(`72d4bf6`),但那是**治标**——本质是"每个共享窗口各起一个 kernel"这个模型不对。
2. **愿景**:用户要"**LAN 内所有 quill(含以后的 Mac)框进一个视图**"(见 ADR-0015 R1「连上同步全部工作区」),手机一处看/控全部。当前"一窗一 kernel"到不了那儿。

设计过程中真正**卡住并最终逼出答案**的是一个小问题:**手机点「+」新建 tab,建到哪里?** 一连串自动启发式被用户逐个用反例击穿(记在下方"被否决的方案"),最终收敛到"共享锚窗口"。

## Decision

### 1. 联邦(超集):机器级单例 kernel + 窗口当 feeder

- **每台机器一个 `quill-kernel` 单例**(独占 `7878`)。多个 quill 窗口**不再各自 spawn kernel**,而是**接入同一个**,各当一个 **feeder**;**每个窗口 = 一个 workspace**。手机看到的是全部 workspace 的聚合。
- **为什么是超集**:多窗口能表达单窗口(N=1),单窗口(强制一窗)表达不出多窗口。多窗口是真需求(多显示器/并排/不同上下文)。**故地基必须是联邦;单窗口是它的免费特例。**(用户原话:"多窗口是单窗口的超集"。)

### 2. 会合(rendezvous):找已有→有则挂、无则建

- 窗口开共享时,经**本机会合点**(本地 unix socket,如 `$XDG_RUNTIME_DIR/quill-kernel.sock` + flock;不用 D-Bus,轻、无依赖、与现有 calloop 一脉相承)判断:
  - **无已有 kernel** → 拉起 kernel、占住 socket/锁,成为 **owner**;
  - **有** → 作为 feeder 接入(把自己的 tab 经 feed 协议喂进去)。

### 3. kernel 生命周期:机器级单例、detach、feeder 引用计数

- **修订 ADR-0018 的"kernel = 发起窗口的子进程"** → kernel 是**机器级单例、detach**(不随发起窗口关闭而死;只要还有 feeder 在共享就活)。
- **feeder 引用计数**:每个接入的窗口 = 一个 feeder;feeder 归零 → kernel 关闭(复用砖0 的 refcount/`destroy_at` 思路,单位从"workspace holder"扩到"feeder")。
- OS 隔离仍在(kernel 是独立进程,崩溃不拖垮任何终端窗口);feeder(窗口)崩溃 → kernel 移除该 workspace。

### 4. ★ home 锚 = 用户开共享的那个窗口 → 新 tab 落这里

- **手机新建 tab 默认落"共享锚窗口"** = **用户亲手点开共享(`--share` / `Ctrl+Shift+S` / 标题栏按钮)的那个窗口**;在联邦里 = **第一个开共享、成为 kernel owner 的窗口**。
- **这是有语义的"#0"**,不是任意的"最早开的窗口"。**开共享这个动作本身就已经指定了 home,零额外 UI/负担**(复用一个用户本来就要做的动作)。
- **一个概念干两件事**:会合里"第一个开共享的窗口 = kernel owner = 锚",这个锚**同时就是 home = 新 tab 落点**。后来 join 的窗口 = feeder(各自 workspace、可寻址、但**非默认新 tab 落点**)。
- **这也是现单窗口行为的自然命名/推广**:现在就一个窗口共享、新 tab(砖2)本来就落它;规则命名为"落点=共享锚窗口"后,1 窗口 → N 窗口 → N 设备平滑推广,不需新机制。
- 换 home = 在别的窗口点开共享(或一个"设为锚"动作);默认即共享窗口,常态不用管。

### 5. 多设备:tab 栏按设备分段;窗口是设备本地细节

- 每台设备各跑自己的 kernel + 对外联邦;**手机 tab 栏按设备分段**(`Linux: [tabs] | Mac: [tabs]`)。加一台设备 = 多一段,线性、无特例。
- **"设备"是联邦的天然节点**,因为 **tab = PTY = 跑在某台设备 OS 上的进程**(不可能设备无关)。"新 tab 在哪台设备开"= 在哪台 OS 起 shell,天然按设备分。
- **同一设备内的多窗口 = 设备本地细节,对联邦不可见**:联邦层是"设备 + 其 tab 聚合";"落哪个本地窗口"由该设备内部按上面的 home 锚决定。远程用户人不在现场、看不见物理显示器,不该也不需要选物理窗口。
- 多设备联邦依赖 **quill-kernel 的 Mac(无头)移植** + 跨设备发现(mDNS,LAN 内);属后续,见 [[project_home_wireguard_vpn_over_v6]] / CLAUDE.md 北极星条。

### 6. 寻址 / 新窗口

- 新 tab 默认 → 锚窗口;想开到别的窗口/设备 → 可选**下钻**到那个 workspace 段点它的「+」(非默认);
- **「+ 新窗口」** = 让 owner 在桌面 `spawn` 一个新 quill 窗口(= 新 feeder / 新 workspace),区别于"新 tab"。

## 保住 E′(为什么不做 session-server)

- **每个窗口仍 own 自己的 PTY、直接渲染(热路径零 IPC、≈今天的 quill、快)**。联邦是**协调 / 看-控中继层**,不是 hot-path 层。
- **不可约**:tab = 进程,进程由窗口拥有;"远程新 tab 落哪个窗口"这个选择删不掉,只能给好默认(= 锚)。
- **唯一能让"窗口"彻底消失**的是"设备跑一个本地 session-server、所有窗口当它的视图" —— 但那把(哪怕本地的)IPC 塞回**每窗口渲染热路径**,正是 ADR-0018 为日用流畅否掉的。**接受"锚窗口默认"这一点点不可约,换 E′ 的速度。**

## 被否决的方案(记 WHY,防重蹈)

- **单实例(新 `quill` 调用 → 现有窗口开 tab,强制一个窗口)**:是多窗口的**子集**,表达不出真多窗口 → 不能当架构主路。**可作为正交、可选的启动便利后加**,不并进本 ADR。
- **session-server(kernel own 所有 PTY、窗口=视图)**:最统一,但热路径回到 IPC → 否(见上)。
- **新 tab 落点的自动启发式,全否**(每个都被用户一个反例击穿):
  - **焦点窗口** → 反例:全屏主窗 + 一个"临时跑命令"的 scratch 窗、焦点在 scratch → 新 tab 不该落 scratch。
  - **#0 / 最早开的窗口** → 可能就是那个 scratch、或已关。
  - **手机当前在看的 workspace** → 你临时看"窗口2的日志"、却想开到主窗 → 目标乱飘。
  - **根因**:"哪个窗口是 home"是个**语义角色(主窗/临时窗/日志窗),无法从焦点/大小/顺序/在看啥推断**。→ 停止猜,让用户用"开共享"这个已有动作**显式且廉价地**指定一次(= §4 锚)。
- **单机全局窗口编号(#0/#1)**:多设备下是负担(机器本位)。→ 用"按设备分段 + 每设备锚"取代。

## Consequences

- ✅ **7878 撞车那类 bug 从架构上消失**(一台机器一个 kernel);ADR-0018 的"启动失败 error 态"退化成仅对**非 quill 进程占 7878** 的兜底(仍保留)。
- ✅ **"新 tab 去哪"有干净、有语义的答案**(锚),且零额外 UI。
- ✅ 1 窗口 → N 窗口 → N 设备**线性扩展、无特例**;兑现 LAN 联邦愿景。
- ✅ E′ 的日用速度(窗口直接渲染、热路径零 IPC)完整保住。
- ⚠️ 代价:**会合 + kernel 生命周期改造(detach + feeder refcount)= 对 E′ 的修订**,是本阶段主要新工。
- ⚠️ web tab 栏要从"单 workspace 平铺"升成"**按设备/workspace 分段**"。

## 分砖(建议顺序,未实现)

- **砖F1 — kernel 单例 + 会合**:本地 socket/flock;第一个开共享 → spawn kernel + 占锁 + owner;后来的 → 接入当 feeder。kernel **detach**(不随发起窗口死)。
- **砖F2 — 多 feeder 汇入**:N 窗口 → N workspace 喂进一个 kernel(复用 `feed.rs`,feeder ≈ 现在的"父→子 tee",只是 N 个父 → 1 kernel);feeder refcount 生命周期;**锚 = 第一个 feeder**。
- **砖F3 — 新 tab → 锚路由**:手机 `TabOp::New` → 锚窗口的 `apply_tab_op`(复用砖2 机制,"父"= 锚 feeder)。
- **砖F4 — web tab 栏分段**:按 workspace(单机)/ 设备(多设备)分组显示;「+」→ 锚;下钻可指定别的 workspace;「+ 新窗口」→ owner spawn 新窗口。
- **砖F5(后)— 多设备联邦**:mDNS 跨设备发现 + 设备分段;依赖 **quill-kernel 的 Mac 无头移植**。

## 非目标 / 推迟

- **单实例(调用变 tab)**:正交、可选,后加,不在本 ADR 关键路径。
- **多设备**:需 Mac(或其它设备)先有 quill-kernel;本 ADR 先把**单机多窗口联邦**做对(F1–F4),多设备(F5)自然接上。
